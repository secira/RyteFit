For an online exam supporting 10,000 concurrent users for 3 hours, here's the recommended architecture:
🏗️ High-Level Architecture
Cloud-Native, Horizontally Scalable Design:
•	Edge Layer: CDN + WAF + Load Balancer
•	Application Tier: Stateless API services (20-40 auto-scaling pods)
•	Data Layer: Aurora PostgreSQL + Redis + S3 storage
•	Real-time Layer: WebSocket service for proctoring
•	Monitoring: Full observability stack

📊 Performance Targets
•	Capacity: 10k concurrent users
•	Autosave: Every 20-30 seconds (≈ 500 writes/sec)
•	Response Times: p95 < 300ms reads, p99 < 1s autosave
•	Uptime: 99.9% availability target


•	🗄️ Database Design
•	PostgreSQL Tables (Partitioned by exam_id):
•	- exams (exam metadata)
•	- exam_instances (user attempts) 
•	- questions (question bank)
•	- exam_questions (shuffled per user)
•	- answers (UPSERT on exam_instance_id + question_id)
•	- autosave_snapshots (incremental saves)
•	- proctor_events (monitoring data)
•	- performance tables to show all India ranking of users
•	- bench mark table to with all India historic data
Core Tables
ExamType
•	ExamTypeID (PK)
•	Name (e.g., NEET 2025 Main, NEET 2025 Mock)
Subject
•	SubjectID (PK)
•	Name (Physics, Chemistry, Biology)
•	OverallWeightage (percentage or marks for this subject)
Chapter
•	ChapterID (PK)
•	SubjectID (FK)
•	Name
•	ChapterWeightage (percentage/marks within subject)
Topic
•	TopicID (PK)
•	ChapterID (FK)
•	Name
Question
•	QuestionID (PK)
•	TopicID (FK)
•	Text
•	Options
•	CorrectOption
 
Weightage Tables
SubjectWeightage
•	SubjectID (FK)
•	ExamTypeID (FK)
•	SubjectWeightage (marks/percentage for this subject in that exam)
ChapterWeightage
•	ChapterID (FK)
•	ExamTypeID (FK)
•	ChapterWeightage (marks/percentage for this chapter in that exam)
 
Relationships
•	Each ExamType can have multiple Subjects.
•	Each Subject contains multiple Chapters.
•	Each Chapter contains multiple Topics.
•	Each Topic contains multiple Questions.
•	Weightage values are stored for both Subjects and Chapters, either at system default or per exam.
 
Example Schema (Markdown Table)
Table Name	Fields/Columns	Description
ExamType	ExamTypeID (PK), Name	NEET/ JEE MAIN
Subject	SubjectID (PK), Name, OverallWeightage	Subject details + weightage
Chapter	ChapterID (PK), SubjectID (FK), Name, ChapterWeightage	Chapter details + weightage
Topic	TopicID (PK), ChapterID (FK), Name	Topic details
Question	QuestionID (PK), TopicID (FK), Text, Options, CorrectOption, DifficultyLevel, PreviousYear, Answers, DetailExplaination 	Questions per topic
SubjectWeightage	SubjectID (FK), ExamTypeID (FK), SubjectWeightage	Subject weightage by exam
ChapterWeightage	ChapterID (FK), ExamTypeID (FK), ChapterWeightage	Chapter weightage by exam

 
Usage Notes
•	To get the weightage for Physics in NEET 2025, check SubjectWeightage for Physics, NEET 2025.[4][2][3]
•	To get chapter-level analysis, check ChapterWeightage for each chapter in an ExamType.[5][1]
•	This design supports rapid tracking, reporting, and analytics of question distribution and study analytics across exams, subjects, and chapters.
 
This schema offers clear subject-level and chapter-level visibility and aligns with NEET's structure and standard practices.[2][3][1]


🎯 Backend Technologies
Primary Choice: FastAPI (Python)
Best for high-scale exams:
•	Performance: Async/await for 10k concurrent connections
•	Auto-docs: Built-in API documentation
•	Type Safety: Pydantic validation prevents bugs
•	WebSocket Support: Real-time proctoring features
•	Easy Deployment: Works with Docker/Kubernetes


🖥️ Frontend Technologies
Primary Choice: Next.js (React)
Best for exam applications:
•	Server-Side Rendering: Fast initial page loads
•	Built-in Security: CSRF protection, secure headers
•	Offline Support: PWA capabilities for connectivity issues
•	Component Library: Rich UI components available
•	TypeScript: Type safety for exam logic

📱 Recommended Stack Combinations
Modern Full-Stack ⭐ Recommended
•	Backend: FastAPI + PostgreSQL + Redis
•	Frontend: Next.js + TypeScript + Tailwind CSS
•	Deployment: Docker + Kubernetes + AWS/GCP

🔧 Key Considerations for 10k Users
Must-Have Features:
•	Database Connection Pooling (PgBouncer)
•	Redis Caching for sessions and questions
•	CDN for static assets
•	Load Balancing across multiple servers
•	Auto-scaling based on user load
•	WebSocket Optimization for real-time features
